# 项目工作规则

- 知识库正文、结构化元数据和索引规则必须保存在仓库内的版本化文件中。
- 新建知识记录时保留来源日期、具体时间、来源位置和原始文件链接；不凭空补充事实。
- 将事实、推断与待确认事项明确区分。
- 不得写入密钥、账号令牌、账号会话、客户敏感正文或 Codex 本地缓存；需要在两台电脑使用的凭据应通过受控密码管理器或企业密钥管理系统分别获取。
- 更新知识后，先检查 `git diff`，再进行小粒度提交并推送。
- 发现另一台电脑已有新提交时，先拉取并处理冲突，禁止使用强制推送覆盖历史。
- 在报告“没有可同步项目”前，默认检查 `D:\Codex\sandbox`；将其中的有效 Git 仓库或带项目清单的目录列为候选项目。候选项目必须先排除凭据、缓存、依赖、构建产物，以及 `outputs/`、`work/` 等生成的 Office 交付物，并检查体积和单文件限制，才能复制到 `projects/` 并推送。skill 内的模板和参考资产不视为输出交付物，可保留。
- 对每个同步项目，先确认并记录“实际源目录 → 仓库目标目录”映射；不得仅根据截图、历史路径或目录名称推断来源。当前 IIJ Workspace 的已确认映射为 `D:\IIJ-Workspace → projects/iij-workspace/`。
- 宣称同步完成前必须完成四阶段验收：源目录文件清单、Git 暂存/跟踪状态、远端分支提交、以及副机拉取后的关键文件清单。`git push` 成功只证明已提交内容抵达远端，不能证明来源完整。
- 对关键文件至少执行 `git status --short --untracked-files=all -- <目标目录>`、`git ls-files -- <目标目录>` 和 `git check-ignore -v -- <关键文件>`；远端地址必须通过 `git remote -v` 确认后再推送。
- 当用户说“推送远端”时：先执行 `git pull --rebase`，检查 `git diff` 与敏感信息，再提交并推送当前机器中可版本化的项目、规则和配置更新；必须排除 `outputs/`、`work/`、密钥、令牌、账号会话及本机缓存。
- 当用户说“拉取远端”时：在当前仓库执行 `git pull --rebase` 并报告结果。若规则或文件出现冲突，停止操作并报告冲突；不得覆盖任一机器已有内容。
