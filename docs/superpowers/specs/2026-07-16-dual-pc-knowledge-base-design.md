# 双 PC 知识库同步设计

## 目标

以本机作为主要工作节点，通过私有 Git 仓库让主机与副机使用同一套可追溯的知识库内容和工作规则。

## 范围

- 将知识条目、元数据、索引说明、项目规则和操作指南纳入 Git 版本管理。
- 为副机提供首次接入与日常同步流程。
- 为大文件和敏感信息提供明确边界。

不包括：创建或授权外部 GitHub/GitLab 账号、同步 Codex 的本地会话/缓存/个人记忆、自动合并冲突。

## 架构

两台 PC 保留各自独立的工作副本。私有 Git 远端仓库是版本化知识的唯一同步中枢；主机和副机通过提交、推送和拉取交换变更。仓库内的 `AGENTS.md` 固化 Codex 的项目规则，`README.md` 记录项目状态和入口，`knowledge/` 保存知识内容。

## 内容边界

| 内容 | 存储与同步方式 |
| --- | --- |
| Markdown、JSON、CSV、脚本、索引及元数据 | Git 追踪 |
| PDF、PPT、图片等大文件 | Git LFS 或受控云盘；知识条目保存链接与摘要 |
| 密钥、令牌、账号会话、Codex 缓存、本地数据库 | 不提交、不直接同步 |
| 未提交修改 | 仅保留在当前电脑，不能视为已同步 |

## 日常数据流

1. 开始编辑前执行 `git pull --rebase`。
2. 更新知识文件后执行 `git add`、`git commit`、`git push`。
3. 另一台 PC 在开始工作或查看最新知识前执行 `git pull --rebase`。
4. 两台电脑同时修改同一条目时，先处理 Git 合并冲突，再继续工作。

## 错误处理与恢复

- 推送被拒绝时，先拉取并解决冲突，禁止用强制推送覆盖另一台电脑的提交。
- 如果误删知识文件，从 Git 历史恢复；大文件依赖对应云盘的版本历史或 Git LFS 历史。
- 每个知识条目必须保留来源时间；若来自客户材料，记录来源位置但不将敏感正文复制到不应存储的位置。

## 验收标准

- 本机是一个可初始化并提交的 Git 项目。
- 项目规则、知识目录说明、忽略规则和副机操作指南齐全。
- 不会追踪凭据、缓存和常见构建产物。
- 副机可仅根据指南完成克隆、检查与同步。
