# 企业 AI 架构指南

## 来源

- 来源：用户在对话中提供的企业 AI、API、RAG、Agent 和治理说明。
- 录入日期：2026-07-07。
- 分类：AI 架构、企业应用、RAG、API 集成、安全治理。

## 概要

企业 AI 不应只被理解为“调用一个大模型”。对客户提案时，应将其设计为一套业务和系统架构：用户界面、应用编排、企业知识库、模型服务、API 集成、权限控制、日志审计和运营流程需要共同设计。

## 典型架构层次

| 层次 | 说明 | 提案关注点 |
|---|---|---|
| 用户层 | 员工、销售、客服、管理者或客户门户 | 使用场景、权限、体验、人工复核 |
| 应用层 | AI 助手、检索问答、摘要、分类、流程触发 | 工作流、异常处理、操作边界 |
| 知识层 | 文档、FAQ、产品资料、客户材料、数据库 | 来源可信度、版本、权限、更新频率 |
| 检索层 | RAG、向量库、关键词检索、Text2SQL、Graph RAG | 检索质量、召回率、引用、可追溯 |
| 模型层 | LLM、Embedding 模型、多模态模型 | 适用任务、成本、延迟、供应商边界 |
| 集成层 | API、Webhook、业务系统连接 | 权限、日志、失败处理、成本控制 |
| 治理层 | 安全、合规、审计、运营、评估 | 数据边界、脱敏、保留期限、人审机制 |

## API 基础说明

API 是系统之间按约定交换能力和数据的接口。企业 AI 项目中，API 常用于：

- 让业务系统调用摘要、分类、问答或生成能力。
- 从 CRM、ERP、文档库或数据库读取必要上下文。
- 将 AI 结果写回工作流或审批系统。
- 记录调用日志、成本、用户和错误。

提案中不要只写“接 API”。需要说明谁调用、调用什么数据、返回什么结果、失败如何处理、日志如何保留、权限如何控制。

## RAG 说明

RAG 是“检索增强生成”。它先从企业批准的知识源中检索相关内容，再把检索结果作为上下文交给模型回答。

适合场景：

- 内部规程、产品资料、FAQ、客户资料、提案库。
- 内容经常更新，不适合频繁训练模型的知识。
- 需要答案引用来源、便于人工验证的场景。

关键控制：

- 知识源是否可靠。
- 文档切分是否合理。
- 检索权限是否继承原系统权限。
- 回答是否展示引用。
- 高风险答案是否需要人工复核。

## AI Agent 说明

AI Agent 不只是聊天机器人，而是可以组合推理、检索、工具调用和工作流动作的系统。

在客户项目中，Agent 只有在以下条件明确后才建议推进：

- 可调用哪些工具。
- 哪些操作必须人工审批。
- 失败或异常时如何停止。
- 日志如何记录。
- 是否可能误操作客户系统或发送错误信息。

## 幻觉风险控制

大模型可能生成看似合理但不准确、缺乏依据或引用错误的内容。不能向客户承诺完全消除幻觉。

实际控制方式：

- 使用 RAG 限制回答依据。
- 展示引用来源。
- 对输出设置结构化格式。
- 对客户对外、高风险或合规相关内容保留人工复核。
- 记录审计日志。
- 对无法确认的信息明确回答“不确定”。

## 部署策略

| 部署方式 | 优点 | 风险/限制 |
|---|---|---|
| 公有云 API | 上线快、模型能力更新快、无需自建模型基础设施 | 需确认数据边界、供应商条款、保留策略和跨境问题 |
| 私有化部署 | 控制边界更清晰，可降低部分外部暴露 | 成本高，仍需治理、补丁、运维和权限设计 |
| 混合架构 | 可按数据敏感度分层处理 | 架构和运营复杂度更高 |

## 提案常见误区

- 只谈模型，不谈业务流程。
- 只谈私有化部署，不谈治理和运营。
- 将 RAG 误说成训练模型。
- 忽略权限继承、日志审计和人工复核。
- 没有说明成本、延迟、调用量和失败处理。
- 对最新法规、产品规格或供应商条款未经确认就对外承诺。

## 客户确认清单

- 要改善的业务流程是什么？
- 哪些知识源可以被 AI 使用？
- 哪些数据不能发送给外部模型？
- 用户权限如何映射到知识权限？
- 是否需要引用来源？
- 哪些输出需要人工复核？
- 是否允许 API 调用外部 LLM？
- 日志保留多久，由谁查看？
- 成本和响应时间目标是什么？
- PoC 成功标准是什么？

## 复用建议

本指南适用于企业 AI 初期咨询、RAG 提案、API 集成说明、客户安全审查回答和内部知识库设计讨论。
