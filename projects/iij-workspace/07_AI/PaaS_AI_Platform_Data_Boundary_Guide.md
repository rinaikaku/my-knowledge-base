## 概要

PaaS AI 平台应清楚区分三个边界：客户数据域、AI 应用平台域、外部 LLM 供应商域。客户拥有的文档、数据库、权限和业务数据应保留在客户或平台控制的环境中。AI 平台只检索当前请求所需的最小上下文，执行权限控制、脱敏、日志和策略检查后，再向外部 LLM 服务发送最小化请求用于推理。

外部 LLM 供应商是否会保存、审核或二次使用 API 输入/输出，取决于合同、服务条款、保留设置和审计证据。不能仅凭“加密”“Token 化”或“通过 API 调用”就断言数据不会被训练。

## 架构阅读方式

典型架构可分为：

- AI 云平台底座。
- 客户专属底座或专属域。
- 用于数据隔离的独立基础环境。
- AI 企业应用底座，包括应用、数据库、Redis、服务实例、LLM 接口、任务编排、备份和服务日志。

## 责任域

### 1. 用户与客户域

包括：

- PC 和移动端用户。
- 客户身份、SSO、IAM、MFA 和 RBAC。
- 企业自有文档、数据库、知识库和业务数据。
- 客户数据访问规则和分级策略。

客户通常负责：

- 决定谁可以访问哪些数据。
- 定义哪些数据可用于 AI 工作流。
- 维护来源文档和业务数据。
- 确认数据分类、保留和删除要求。
- 批准业务场景和风险等级。

### 2. AI 应用平台域

包括：

- AI 应用前端和后端。
- RAG 检索。
- 企业知识库访问。
- DB 和 Redis。
- 向量数据库或向量索引。
- 服务实例。
- LLM 接口或 AI Gateway。
- 任务编排。
- 备份和服务日志。
- 审计、监控和运营。

平台通常负责：

- 认证用户并执行授权。
- 只检索被允许访问的数据。
- 最小化发送给 LLM 的数据。
- 尽可能脱敏敏感信息。
- 防止 Prompt Injection 和未授权工具调用。
- 将请求路由到已批准模型。
- 按策略记录审计日志。
- 管理平台可用性、服务运营和事件响应。

### 3. LLM 供应商域

包括：

- 托管模型推理服务。
- 模型 API。
- 供应商侧日志、监控和安全控制。

供应商通常负责：

- 接收已准备好的 Prompt 或请求负载。
- 使用托管模型进行推理。
- 返回生成结果。
- 按合同执行数据处理承诺。
- 对训练、保留、人工审核和二次使用执行合同限制。
- 在适用时提供服务文档、审计材料和合规承诺。

## 数据流边界

### 应保留在客户或平台域内的内容

- 完整企业文档库。
- 完整客户数据库。
- 完整用户身份和权限系统。
- 完整 RAG 知识库。
- 向量数据库内容，除非明确需要发送。
- 内部服务日志和审计日志。
- 备份数据。
- 推理不需要的原始敏感数据。

### 可能发送给 LLM 的内容

- 用户问题。
- 被选中的 RAG 片段。
- 必要结构化上下文。
- 系统指令。
- 输出格式要求。
- 可行时经过脱敏或假名化的字段。

### 需要向客户解释清楚的点

- RAG 上下文是当前推理的临时上下文。
- RAG 上下文不会自动成为模型训练数据。
- 通过 API 发送数据本身，并不等于模型正在被训练。
- 但是，供应商是否保存、人工审核或复用数据，取决于服务条款和配置。

## 如何证明“数据不用于模型训练”

对客户不能只做口头承诺，应准备以下证据：

- API 输入/输出数据不用于模型训练的合同或条款。
- API 输入/输出数据不用于模型改进的条款。
- 不做服务交付之外的二次使用。
- 明确保留期限。
- 明确删除政策。
- 明确人工审核政策。
- 明确滥用监控范围。
- 明确租户或逻辑隔离。
- 明确分包商和跨境传输条款。

## 平台侧设计建议

- 将来源文档和数据库保留在客户或平台控制域内。
- 每次请求只检索必要片段。
- 外部 LLM 调用前进行敏感数据脱敏。
- 只发送最小化 Prompt 和上下文。
- 使用 LLM Gateway 做模型路由和策略执行。
- 防止 LLM 直接访问完整客户资料库。
- 记录审计所需元数据，避免过度暴露敏感内容。

## 加密不等于“不训练”

加密主要保护：

- 传输中的数据。
- 存储中的数据。
- 对存储数据的访问。

但加密本身不能回答：

- 供应商是否保存 Prompt 或回答内容。
- 保留内容是否用于评估。
- 数据是否用于 Fine-tuning。
- 数据是否由人工审核。
- 数据是否进入任何模型改进流程。

客户对外解释：

```text
加密可以降低传输和存储过程中的泄漏风险，但不能单独证明数据不会被用于模型训练。是否用于训练或模型改进，需要以供应商合同条款、数据保留设置和审计证据确认。
```

## Token 化或编码不等于“不训练”

- Tokenization 会把文本转换成模型可读取的单元。
- 假名化可以降低直接识别风险。
- 脱敏可以移除或隐藏敏感字段。
- 这些技术降低暴露风险，但不会自动阻止供应商侧保留或二次使用。

客户对外解释：

```text
Token 化、编码或脱敏是降低数据暴露的技术措施，但不能替代合同和平台配置层面的“不用于训练”证明。
```

## RAG 与训练边界

| 项目 | RAG | 训练 / 微调 |
|---|---|---|
| 目的 | 使用检索上下文回答当前问题 | 改变模型参数或行为 |
| 数据使用时间 | 当前请求 | 训练流程 |
| 更新方式 | 更新知识库或索引 | 准备训练数据并重新训练 |
| 风险重点 | 权限、引用、检索质量、上下文泄漏 | 数据进入训练集、模型记忆、二次使用 |

```text
该架构将客户数据域、AI 应用平台域和外部 LLM 供应商域分开管理。客户文档、数据库、权限、日志和知识库保留在客户或平台控制环境中。AI 平台只检索当前请求所需信息，执行访问控制、脱敏和策略检查后，向 LLM 推理 API 发送最小化 Prompt。LLM 供应商返回生成结果，API 输入/输出不应被用于模型训练或模型改进；这一点需要通过供应商条款、DPA、保留设置和审计证据确认。
```

## 提案检查清单

- 谁拥有客户数据。
- 谁运营 AI 应用平台。
- 使用哪些 LLM 供应商。
- 每个供应商合同是否写明不训练或不用于模型改进。
- Prompt 和回答数据是否保留。
- 是否支持零数据保留。
- 是否允许供应商人工审核。
- 数据是否跨地区或跨境。
- 是否保存完整 Prompt/回答日志。
- LLM 调用前是否实施 DLP、脱敏和最小化。
- RAG 来源权限是否执行。
- 架构图是否清楚区分客户域、AI 平台域和 LLM 供应商域。

## 复用建议

本指南适用于企业 AI PaaS 架构客户说明、数据隔离和责任边界提案、安全审查答复、外部 LLM API 使用 FAQ，以及需要明确用户域、平台域和供应商域的架构图。
