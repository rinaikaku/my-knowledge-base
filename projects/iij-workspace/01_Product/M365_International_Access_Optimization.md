# M365 国际版中国访问优化

**分类：** 产品与方案 / 网络与云访问  
**适用范围：** 中国站点访问 Microsoft 365 国际版的网络设计  
**知识状态：** 可复用；落地前必须进行客户现场 PoC  
**更新日期：** 2026-07-15

## 核心结论

推荐采用 **PAC 精准分流 + M365 专用安全出口 + 上海至东京 IPLC + 日本侧 DIA/Microsoft 网络互联** 的组合。其目标是让 Microsoft 365 流量与普通互联网流量分治，减少通用显式代理的性能与兼容性影响，并保留企业所需的安全和运维控制。

## 推荐流量模型

1. 客户终端通过 PAC 获取流量规则。
2. Microsoft 365 已识别端点流量进入专用 NGFW，按 Microsoft 官方端点类别与 TLS 检查例外策略处理后，经 IPLC 至日本侧 DIA/IIJ 互联网出口，再访问 Microsoft 全球网络。
3. 非 Microsoft 365 的互联网流量仍通过企业代理或通用 NGFW，使用中国本地 DIA 出口。
4. 管理面保留身份认证、日志留存、监控、配置备份、跳板机 MFA 与会话审计、漏洞扫描和基线加固。

## 设计要点

- 不以固定 IP 列表识别 M365：应使用 Microsoft 365 官方端点服务与分类信息，建立定期同步、变更审核和 PAC/防火墙规则更新机制。
- 对 Teams 等实时流量评估时，除时延外还需记录丢包、抖动、可用性和会议质量指标。
- 专用路径不等于放弃安全：需要明确出口控制、最小必要的 TLS 解密例外、DNS 策略、日志关联和终端安全责任。
- 路由、Anycast 落点和 Microsoft 服务端点均可能变化；历史测试数据仅可作为方案方向依据，不能替代当前 PoC。
- 跨境线路、日志、身份数据和业务数据需按客户合规政策与适用法律确认。

## 术语与实施澄清

### DIA（Dedicated Internet Access）

DIA 通常指企业互联网接入/出口能力，不是 M365 自带服务，也不是每一条跨境专线天然包含的组成部分。实际合同形态取决于运营商与方案：可单独采购，也可由服务商以“专线 + 日本侧互联网出口”或托管网络服务的组合方式交付。项目设计时必须确认出口所在地、带宽、IP/路由、SLA、运营责任和费用是否已包含在回线服务中。

### Microsoft MSN / AS8075 与运营商路径

Microsoft MSN（AS8075）可理解为微软自建的全球骨干网络。M365 访问优化的重点是使用户流量以较短、稳定的路径进入 Microsoft 网络，而不是假定服务固定部署在某一城市的单一服务器上。资料中出现的中国电信、中国联通、PCCW、IIJ 等名称，表示流量在运营商网络之间的交接或承载边界，不是完整设备跳数。不同运营商的对等互联与 BGP 路由不同，不能仅按“节点数量”判断体验优劣，应以当前环境的业务测试结果为准。

### Capex 转 Opex

将 DNS、虚拟化或安全设备等实体设备的购买与周期性更换，转换为云服务或托管服务的持续使用费。其主要价值是降低前期资金占用、减少 EOL 替换压力并转移部分运维责任；并不天然意味着长期总成本更低，项目中仍需比较合同期限、扩容、退出迁移和服务商依赖风险。

### Microsoft 365 官方端点分类

“订阅端点分类”不是 M365 管理中心内的付费订阅功能。Microsoft 对外提供 Endpoint Web Service、下载数据和 RSS 变更通知，供网络团队获取 URL、IP、端口、服务区域及 `Optimize`/`Allow`/`Default` 类别，再经客户审批同步到 PAC、代理、防火墙或 SD-WAN 规则中。实施不应把自动获取的数据未经审核直接推送到生产设备。

### PoC 的三项网络指标

| 指标 | 大白话 | 对用户的典型影响 | 如何验证 |
| --- | --- | --- | --- |
| 时延（Latency/RTT） | 一次请求来回要多久 | 语音对话有明显停顿、操作响应慢 | 在相同用户位置、相同业务时段，对新旧路径记录 Teams UDP 时延和 Microsoft 365 网络测试结果 |
| 抖动（Jitter） | 每个数据包到达时间是否忽快忽慢 | 声音断续、漏字、画面忽清忽糊 | 连续采样 Teams 通话的 UDP 抖动；与会议质量记录关联 |
| 丢包（Packet Loss） | 发出的数据包有多少没有到 | 音频卡顿、机器人音、视频马赛克或掉线 | 连续采样 Teams UDP 丢包率，并排除 Wi-Fi 覆盖、终端性能和本地拥塞干扰 |

建议使用 Teams 管理中心 Call Quality Dashboard（CQD）或 Microsoft 365 网络连接测试，配合出口设备/线路监控，在变更前后对同一用户群做至少一周的工作时段基线对比。不能只用一次 `ping` 结果判断 Teams 质量。

## 适用与不适用

**适用：** 对 Outlook、Teams、SharePoint、OneDrive 等国际版 M365 的体验、稳定性或运维可观测性有较高要求的中国站点。

**需要另行评估：** 仅追求最低成本、无法提供跨境专线、禁止特定跨境数据路径、或 M365 流量与其他 SaaS 无法可靠分离的场景。

## 验证清单

- 盘点现网代理、PAC、DNS、出口、安全检查和 M365 端点规则。
- 使用代表性用户和真实业务时段进行 PoC；覆盖 Exchange、Teams 媒体、SharePoint/OneDrive。
- 建立变更前基线与变更后指标：时延、抖动、丢包、登录成功率、Teams Call Quality、可用性。
- 验证回退路径、线路故障切换、日志完整性、告警和运维交接。
