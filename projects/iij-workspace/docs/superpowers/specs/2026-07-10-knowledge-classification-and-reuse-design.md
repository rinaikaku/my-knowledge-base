# 知识分类与复用机制设计

## 目标

将 `D:/IIJ-Workspace` 作为可持续复用的业务知识库。无论资料来自客户材料、会议纪要、邮件、产品说明、PPT 还是 AI 对话，都应先区分知识归属，再沉淀到合适位置，并能在新项目中按主题、安全边界和适用场景快速调用。

本设计覆盖历史条目的渐进治理和未来条目的统一录入，不要求一次性重写全部既有文件。

## 已有基础

- 目录已经按产品、FAQ、提案、邮件、客户、SOP、AI、合规和索引划分。
- `06_SOP/Knowledge_Intake_SOP.md` 已定义基础录入流程、敏感信息边界和来源时间保留要求。
- `99_Index/Main_Index.md` 与 `99_Index/Product_Index.md` 已作为主要导航入口。

## 方案比较

### 方案一：只依赖目录和文件名

优点是维护成本最低；缺点是资料增多后难以判断来源、适用对象、可复用程度和时效性，不建议采用。

### 方案二：建立独立数据库或知识管理平台

优点是筛选和统计能力强；缺点是需要额外系统、迁移和维护成本。当前工作区以 Markdown 为主，暂不作为第一阶段方案。

### 方案三：在现有 Markdown 结构上增加轻量元数据和复用流程

保留现有目录与文件，新增统一的条目头部字段、历史条目治理状态、专题入口和调用规则。成本低、可逐步落地，推荐采用。

## 信息架构

### 知识归属

| 知识类型 | 存放位置 | 典型内容 | 复用边界 |
|---|---|---|---|
| 通用产品与解决方案 | `01_Product` | 服务定位、范围、差异、交付物、适用场景 | 可用于多客户提案；外部数据需复核 |
| 高频问答与异议 | `02_FAQ` | 客户问题、回答逻辑、边界说明 | 可复用；不得包含客户敏感信息 |
| 提案逻辑与话术 | `03_Proposal` | 价值主张、章节结构、路线图、案例表达 | 可按客户修改；必须标记假设 |
| 邮件模板 | `04_Email` | 可复用邮件结构与措辞 | 不保存一次性敏感内容 |
| 客户专属信息 | `05_Customer` | 背景、需求、角色、进展、行动项 | 仅限对应客户项目使用 |
| 可重复流程 | `06_SOP` | 录入、调查、交付、项目推进流程 | 内部复用 |
| AI 方法与知识 | `07_AI` | AI 架构、RAG、提示词、数据边界 | 通用复用；对外主张需复核 |
| 合规知识 | `08_Compliance` | 法规、标准、评价体系、合规话术 | 必须记录来源与复核要求 |
| 导航与专题入口 | `99_Index` | 主索引、分类索引、专题页 | 仅维护入口，不堆积正文 |

### 条目最小元数据

新建条目或实质性更新条目时，建议在标题后补充以下字段；不适用的字段可省略：

```markdown
## 条目信息

- 分类：产品 / 客户 / 提案 / FAQ / SOP / AI / 合规
- 适用场景：可复用场景描述
- 来源：文件、会议、邮件或用户提供资料
- 来源日期：YYYY-MM-DD（来源包含时填写）
- 更新日期：YYYY-MM-DD
- 复用级别：通用 / 受限 / 客户专属
- 对外使用前复核：是 / 否；说明需核验内容
```

字段含义：

- **通用**：可在不同客户或项目中复用。
- **受限**：可作为内部参考或匿名化案例，不得直接对外复述。
- **客户专属**：仅供指定客户或项目使用。

## 历史条目治理方式

### 原则

- 不因格式统一而重写无关正文。
- 不删除既有信息，不补造来源或日期。
- 原材料已有日期和具体时间时，保留其原始时间信息。
- 优先治理经常被调用、即将用于对外输出或内容边界不清的文件。

### 治理等级

| 等级 | 触发条件 | 处理动作 |
|---|---|---|
| A：立即治理 | 即将用于客户材料、存在合规或敏感信息、主题混杂 | 补充元数据、来源、复用级别、待核验事项和索引 |
| B：调用时治理 | 已有基本结构，但后续项目再次使用 | 在本次调用时补齐缺失字段和入口 |
| C：保留待整理 | 暂未使用且风险较低 | 不做批量改写，保留至后续调用时处理 |

## 新资料录入流程

1. 读取来源资料，提取主题、日期、具体时间、事实、假设和敏感信息。
2. 判断资料属于通用知识、客户专属知识或两者兼有。
3. 将通用部分沉淀到相应分类目录；客户事实仅写入 `05_Customer` 或项目专属目录。
4. 对涉及外部统计、法规、价格、产品规格和工期的内容，标记对外使用前复核要求。
5. 新建或更新条目时补充最小元数据。
6. 更新相应索引或专题入口。
7. 校验 Markdown 链接、中文输出和敏感信息边界。

## 新项目调用流程

1. 明确项目对象、服务对象和资料使用边界。
2. 从 `99_Index/Main_Index.md` 进入对应专题，检索产品、案例、FAQ、提案与合规知识。
3. 仅调用复用级别为“通用”的条目；“受限”条目需先匿名化；“客户专属”条目不得跨客户调用。
4. 将已调用内容与新项目事实分开组织，明确哪些是来源事实、哪些是推断或方案建议。
5. 对外输出前复核可能变动的信息，并确认客户范围、报价范围和数据边界。

## 统一调用指令

用户可使用以下表达触发知识库检索与复用：

```text
新建[客户名称]项目：参考工作区中与[产品/主题]相关的通用知识，生成提案大纲；客户专属资料不得跨客户引用。
```

```text
根据以下资料更新知识库：请区分通用知识、客户专属信息、FAQ、提案话术和合规事项；保留来源日期；更新对应索引。
```

```text
为[客户名称]准备[邮件/提案/会议纪要]：调用工作区已有的[主题]内容，并列出对外使用前需要确认的事项。
```

## 实施范围

第一阶段实施以下内容：

1. 更新入门说明和知识录入 SOP，加入最小元数据、复用级别和历史条目治理规则。
2. 新增知识库使用规范页面，并加入主索引。
3. 为当前已纳入索引的条目建立轻量治理清单，标记 A、B、C 级，不批量重写正文。
4. 选取近期新增的“中国据点安全调查服务”条目作为 A 级样例，补充统一元数据。

不在第一阶段实施的内容：

- 不引入数据库、向量检索服务或额外软件。
- 不批量改写所有历史条目。
- 不将客户专属信息迁移到通用目录。

## 验收标准

- 新资料可以按统一分类写入对应目录。
- 每个新建或实质更新条目可识别其来源、适用场景和复用边界。
- 新项目可以从主索引定位并调用相关通用知识。
- 客户专属资料不会被作为跨客户通用材料使用。
- 对外易变内容具有明确的复核提醒。
- 历史条目治理采用按需补齐，不造成无关内容重写。

## 风险与控制

| 风险 | 控制措施 |
|---|---|
| 资料重复沉淀 | 每个条目只保留一个主要主题，索引只维护入口。 |
| 客户信息跨项目泄露 | 使用复用级别字段；客户专属内容只存放在客户目录。 |
| 过期法规或产品信息被直接引用 | 使用“对外使用前复核”字段并在交付前复查。 |
| 历史资料一次性整理成本过高 | 使用 A/B/C 分级，按调用频率与风险渐进治理。 |
